# Contributing

This guide covers local dev setup, the two most common contribution shapes (adding a backend, writing a skill), testing conventions, and how to ship a PR.

For the bigger picture of how the pieces fit together, see [architecture](./architecture.md) first.

## Dev setup

```bash
git clone https://github.com/brandtam/rubber-ducky.git
cd rubber-ducky
pnpm install           # or npm install
pnpm build             # or npm run build
pnpm link --global     # or npm link
```

`pnpm`, `npm`, and `yarn` all work. The project uses TypeScript and ships compiled `.js` to `dist/`.

### Day-to-day commands

```bash
pnpm dev -- <command>       # run the CLI via tsx (no build step)
pnpm test                   # full test suite
pnpm test:watch             # vitest watch mode
pnpm typecheck              # tsc --noEmit
pnpm build                  # compile to dist/
```

`pnpm dev` lets you iterate on CLI changes without rebuilding:

```bash
pnpm dev -- status
pnpm dev -- ingest asana TIK-4647
```

## Adding a new backend

A backend is anything you want to ingest tasks from or push tasks to — Slack threads, Linear tickets, Notion databases, etc. Every backend follows the same shape so skills and CLI commands can treat them uniformly.

### The `Backend` interface

Defined in `src/lib/backend.ts`. Every backend implements:

```ts
interface Backend {
  name: string;                           // "github", "jira", "asana", etc.
  capabilities: Capability[];             // which operations are supported
  ingest(ref: string): Promise<TaskPage>;
  pull(taskPage: TaskPage): Promise<PullResult>;
  push(taskPage: TaskPage): Promise<PushResult>;
  comment(taskPage: TaskPage, text: string): Promise<CommentResult>;
  transition(taskPage: TaskPage, status: Status): Promise<TransitionResult>;
}
```

Declare `capabilities` honestly — `assertCapability()` will throw a clear error if a skill calls an unsupported operation, which is preferable to silently no-oping.

### The shape of a new backend

For a new backend called `linear`, you'd typically add:

- `src/lib/linear-client.ts` — thin REST client. Takes a fetch injector for testability. Knows nothing about `TaskPage` or rubber-ducky's domain model — it just speaks the external API.
- `src/lib/linear-backend.ts` — implements `Backend`. Wraps the client, maps external fields to `TaskPage`, maps wiki `Status` values to backend-specific statuses.
- `src/lib/linear-ingest.ts` — uses `ingest-shared.ts` for dedup, page generation, concurrency, and attachment handling. This is the same shared surface every backend's ingest uses.
- Registration in `src/lib/backend.ts` — add a case to `getBackend()` so workspace config can resolve to a backend instance.
- Tests — see [Testing conventions](#testing-conventions).

### The shared ingest layer

`src/lib/ingest-shared.ts` provides the common ingest plumbing: dedup (skip tasks already in the wiki), page generation (consistent frontmatter shape), concurrency (bounded parallelism for bulk ingests), and attachment downloading into `raw/assets/`. New backends should route through it rather than reimplementing page creation.

### Register a backend skill

Bundled skills that know how to invoke your backend from Claude Code live in `src/lib/templates.ts` under `generateBackendSkills()`. Add a case for your new backend type that returns a skill markdown blob (similar to `generateIngestAsanaSkill`). The skill is installed automatically when a user configures your backend during `init`.

## Writing a skill

Skills are plain markdown files in `.claude/commands/`. They become slash-commands in Claude Code (`/my-skill`) and are discovered automatically.

### Shape of a skill file

```markdown
# My Skill

One-line description of what this skill does.

## Behavior

Step-by-step instructions Claude Code follows when invoked. Reference
CLI commands with fenced code blocks:

\`\`\`
rubber-ducky page create task "..."
\`\`\`

## Output

What Claude should produce at the end.
```

There's no schema enforcement — skills are prompts. But consistent structure (`Behavior` → `Output`) makes them readable and maintainable.

### Bundled vs. developer-only skills

- **Bundled skills** ship to every user workspace via `init` and `rubber-ducky update`. They live as string literals in `src/lib/update.ts` (`getBundledTemplates()`). Add one by appending to the returned array.
- **Backend-specific skills** are installed only when the user configures a given backend. They're generated by `src/lib/templates.ts` (`generateBackendSkills()`).
- **Developer-only skills** (like `/commit`, `/write-pr`, `/write-a-prd`) live at the repo root's `.claude/commands/` and are used by contributors working on rubber-ducky itself. They don't ship to user workspaces.

Where a new skill lands depends on its audience.

### When to use an agent vs. a skill

Agents (`.claude/agents/`) are specialized sub-processes with narrow tool allow-lists — use them when a task needs focused reasoning over a constrained tool surface (e.g., read-only queries, structured output). Skills invoke agents when appropriate. Don't wrap everything in an agent; the overhead isn't worth it for one-shot operations.

## Testing conventions

Tests live in `src/__tests__/` and use [vitest](https://vitest.dev/).

**Good tests check external behavior, not implementation details.** A CLI command's test should invoke the command and assert on stdout, file contents, or exit code — not on which helpers were called along the way.

### Test layout

- `<module>.test.ts` — unit tests for a lib module (pure functions, classes with injected dependencies).
- `<module>-cli.test.ts` — end-to-end CLI tests that spawn the compiled CLI and assert on output.

Examples to read for patterns:

- `asana-backend.test.ts` — backend tests with an injected fetch mock.
- `backend-cli.test.ts` — CLI tests for `backend list` and `backend check`.
- `configure-naming.test.ts` — interactive-prompt tests with a stubbed readline.
- `linter.test.ts` — wiki-state tests with an in-memory workspace fixture.

### Dependency injection for testability

Modules that touch the outside world accept their dependency via a parameter:

- `AsanaClient` takes a `fetch` injector.
- `JiraClient` takes a `fetch` injector.
- CLI commands that spawn subprocesses take an `exec` injector.

This pattern means tests never hit a real Asana/Jira server or shell out to `gh`.

### Running tests

```bash
pnpm test                   # run everything
pnpm test asana             # filter by name
pnpm test:watch             # watch mode for TDD
pnpm typecheck              # TS only, no runtime
```

The RGR loop (Red → Green → Refactor) works well here: write a failing test, make it pass with the simplest implementation, then refactor without changing behavior.

## Submitting a PR

1. **Branch from `main`** with a descriptive name:

   ```bash
   git checkout -b feature/linear-backend
   ```

2. **Keep commits atomic.** One commit = one reviewable change. The `/commit` skill generates structured messages from the current diff.

3. **Make sure tests and typecheck pass** before opening the PR:

   ```bash
   pnpm typecheck && pnpm test
   ```

4. **Open the PR.** The `/write-pr` skill drafts a description from your branch diff — use it or write one manually with a Summary and Test plan section.

5. **Link to any related issues** with `Closes #NN` or `Fixes #NN` so they auto-close on merge.
